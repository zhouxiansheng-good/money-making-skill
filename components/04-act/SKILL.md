# 04-act：行动层

> **"实践是检验真理的唯一标准"** — 不验证就不算完成。

## 职责

设计MVP验证实验，快速迭代，从服务到产品升级。核心原则：**最小成本验证需求，最快速度获得反馈**。

## 前置条件

- 杠杆层（03-leverage）已确定杠杆策略
- 评估层已输出Go决策

## 组件

### 1. MVP设计器

**目标**：设计最小验证实验，用最低成本验证需求存在。

**MVP设计流程**：

```
Step 1: 明确核心假设
  "这个赚钱机会成立的核心假设是什么？"
  例: "有人愿意为AI生成的种草文案付费"

Step 2: 设计最小验证
  "用最低成本验证这个假设的方法是什么？"
  例: "在闲鱼发3个AI文案服务商品，看有没有人咨询"

Step 3: 定义成功标准
  "怎么算验证通过？"
  例: "7天内获得≥3个付费咨询"

Step 4: 估算成本
  "验证需要投入多少时间和金钱？"
  时间: ≤2小时/天 × 7天 = 14小时
  金钱: ≤100元（平台费用）

Step 5: 设定时间盒
  "验证周期多长？"
  ≤2周
```

**MVP类型选择**：

| MVP类型 | 适用场景 | 成本 | 验证速度 |
|---------|----------|------|----------|
| 伪装门面 | 验证需求是否存在 | 极低 | 极快 |
| 单功能原型 | 验证核心功能是否有价值 | 低 | 快 |
| 服务化MVP | 先做服务验证，再产品化 | 低 | 中 |
| 预售MVP | 先收钱再做 | 极低 | 极快 |

**MVP设计原则**：
- 不追求完美，追求验证
- 每个MVP只验证1个核心假设
- 时间盒≤2周
- 成本≤200元

### 2. 快速迭代器

**目标**：验证→调整→再验证，每轮迭代≤1周。

**迭代循环**：

```
┌──────────────────────────────────┐
│                                  │
│  Plan → Do → Check → Act        │
│    ↑                    ↓        │
│    └──── 学习 ←─────────┘       │
│                                  │
└──────────────────────────────────┘

Plan: 设计本轮验证实验
Do:   执行实验
Check: 检查结果是否达到成功标准
Act:   根据结果决定下一步
  - 达标 → 进入下一阶段或扩大规模
  - 未达标但接近 → 调整参数，再验证
  - 未达标且差距大 → 回到评估层重新筛选
```

**迭代决策规则**：

| 结果 | 决策 | 行动 |
|------|------|------|
| 达标 | 放大 | 增加投入，扩大规模 |
| 接近达标（差距<30%） | 调整 | 修改1-2个参数，再验证1轮 |
| 远未达标（差距>30%） | 转向 | 回到评估层，换一个方向 |
| 连续2次远未达标 | 放弃 | 回到评估层重新筛选机会 |

**3次失败规则**：连续3次MVP验证失败 → 回到思维层重新审视特定知识。

### 3. 产品化升级器

**目标**：从服务→系统→产品，实现可规模化。

**升级路径**：

```
Level 1: 服务（时间换钱，但验证需求）
  ├─ 1对1服务（咨询/代写/定制）
  ├─ 验证有人愿意付费
  └─ 积累案例和口碑

Level 2: 系统（部分自动化，效率提升）
  ├─ 将服务流程标准化
  ├─ 用AI工具提升效率
  └─ 1对多服务（课程/模板/工具包）

Level 3: 产品（完全自动化，可规模化）
  ├─ 将系统产品化（软件/SaaS/自动化工具）
  ├─ 边际成本趋近于零
  └─ 收入不与时间挂钩
```

**升级判定标准**：

| 升级条件 | 判定标准 |
|----------|----------|
| 服务→系统 | 累计服务≥10个客户，流程可标准化 |
| 系统→产品 | 系统运行≥1个月，客户反馈稳定，需求可产品化 |

**关键原则**：
- 不跳级：没验证需求就不做产品
- 先卖再做：先有人付费，再投入开发
- 小步快跑：每次升级只做最小必要变更

## 产出

- MVP验证计划（假设+验证方法+成本+时间+成功标准）
- 迭代记录（每轮的Plan-Do-Check-Act）
- 产品化升级路径（当前级别+下一步升级条件）

## 路由

- MVP验证通过 → 扩大规模或进入产品化升级
- MVP验证失败 → 回到 `components/02-evaluate/SKILL.md` 重新筛选
- 连续3次失败 → 回到 `components/01-think/SKILL.md` 重新审视特定知识
